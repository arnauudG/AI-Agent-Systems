# Diagramming Rules

## Principle
The agent should prefer visual explanation when diagrams improve clarity, alignment, review quality, or decision-making.

Data work often becomes clearer when people can see:
- sources
- transformations
- outputs
- ownership boundaries
- lifecycle stages
- dependencies

## When to propose or create a diagram
Create or suggest a diagram when:
- the flow crosses multiple systems
- multiple transformations or serving stages exist
- ownership is unclear
- lineage matters
- trust or access boundaries matter
- the audience needs a quick shared understanding
- an MVP or PoC must be explained fast
- architecture options are being compared

## Core rule
Choose the smallest diagram that explains the problem well.
Do not start with maximum complexity.

## What a useful diagram should do
A useful diagram should:
- clarify the main flow
- clarify responsibilities or boundaries
- support a decision or review
- improve shared understanding quickly

## What a diagram should not do
A diagram should not:
- attempt to show everything
- bury the main story in decorative clutter
- exist only to look sophisticated
- replace clear statements of assumptions, trade-offs, and limitations

## Output expectation
When a diagram is appropriate, the agent should ideally provide:
1. the chosen diagram type
2. a short explanation
3. diagram source or a structured diagram brief
4. assumptions and omissions
