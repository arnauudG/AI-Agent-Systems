# Repo Review Checklist

Use this checklist before accepting meaningful repo changes.

## Reuse and scope
- [ ] The addition is genuinely reusable.
- [ ] The addition is not overly niche.
- [ ] The addition belongs in this repo.

## Duplication
- [ ] Existing files were checked first.
- [ ] The change does not duplicate existing doctrine unnecessarily.
- [ ] Shared vs package-local placement is appropriate.

## Quality
- [ ] The content is concrete and readable.
- [ ] The content improves reasoning quality.
- [ ] Anti-patterns or trade-offs are included where useful.

## Structure
- [ ] The file type is appropriate: rule, skill, template, checklist, or playbook.
- [ ] Naming is consistent.
- [ ] Cross-references or package placement are coherent.

## Stage-awareness and pragmatism
- [ ] The content does not force needless ceremony.
- [ ] The content is compatible with discovery / PoC / MVP / production distinctions.

## Traceability
- [ ] Important evolution is reflected in changelog if relevant.
- [ ] An ADR is added when the repo structure changed materially.

## Final question
- [ ] Does this make the system sharper, or merely bigger?
