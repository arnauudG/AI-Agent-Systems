# Diagram Checklist

Use this checklist before considering a diagram “good enough.”

## Purpose
- [ ] The purpose of the diagram is clear.
- [ ] The diagram matches the decision or explanation needed.
- [ ] The diagram matches the intended audience.

## Scope
- [ ] The scope is neither too broad nor too narrow.
- [ ] Only relevant components are shown.
- [ ] Unnecessary detail was removed.

## Structure
- [ ] Main actors / components are identifiable quickly.
- [ ] Important boundaries are shown.
- [ ] Important relationships or flows are visible.
- [ ] The visual structure is readable without narration.

## Clarity
- [ ] Labels are understandable.
- [ ] Naming is consistent with documentation and implementation.
- [ ] Important arrows or flows are directional and meaningful.
- [ ] The diagram avoids visual clutter.

## Decision support
- [ ] The diagram helps explain trade-offs, structure, flow, or ownership.
- [ ] The diagram helps review risk or complexity where relevant.
- [ ] The diagram makes the system easier to discuss.

## Completeness relative to purpose
- [ ] Security boundaries are shown if relevant.
- [ ] Deployment/runtime boundaries are shown if relevant.
- [ ] Ownership or responsibility is shown if relevant.
- [ ] Data movement is shown if relevant.

## Final question
- [ ] Does this diagram genuinely improve understanding, or is it just technical wallpaper?
