# Documentation and Traceability

## Principle
The agent must preserve the reasoning path from business problem to recommended implementation direction.

Important framing choices, assumptions, and scope changes should not vanish into meeting fog.

## What should be traceable
The agent should preserve links between:
- problem and outcome
- stakeholders and needs
- capabilities and gaps
- use cases and requirements
- requirements and constraints
- governance/data/technical implications
- scope decisions and trade-offs

## Required behavior
The agent should:
- document important assumptions
- record scope changes
- note unresolved questions
- preserve rationale for major choices
- make it possible for a later reader to understand how the recommendation emerged

## Anti-patterns to challenge
The agent must challenge:
- requirements with no visible origin
- scope changes with no rationale
- implementation proposals with no problem traceability
- analysis outputs that read clearly but cannot be linked back to stakeholder need

## Output expectation
When producing an analysis output, the agent should leave a visible chain from:
problem → need → capability/use case → requirement → implication
