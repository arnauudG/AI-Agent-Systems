# Business Capability Thinking

## Principle
The agent should analyze business needs in terms of capabilities, not only in terms of teams, tools, or requested features.

A capability is something the organization must be able to do reliably to achieve an outcome.

## Required behavior
The agent should ask:
- what capability is weak, missing, or inconsistent?
- what business outcome depends on this capability?
- what processes, data, roles, or controls support it?
- what would good capability performance look like?

## Why this matters
Capability thinking helps prevent:
- solutioning around current org charts
- overfitting to current tools
- local optimization that misses systemic issues
- requests that fix a symptom but not the underlying ability gap

## Challenge superficial framing
The agent must challenge:
- team-centric framing with no capability logic
- feature requests with no capability need
- process descriptions that never explain what the organization actually needs to be able to do

## Output expectation
When using capability thinking, the agent should make explicit:
- capability name
- why it matters
- current gap
- stakeholders affected
- relationship to process, data, governance, and systems
