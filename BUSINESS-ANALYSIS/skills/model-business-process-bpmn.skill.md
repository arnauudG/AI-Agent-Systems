# Skill: Model Business Process BPMN

## Purpose
Model a business process in a BPMN-oriented way so that actors, handoffs, decisions, and responsibilities become explicit and usable for downstream design.

## Use when
Use this skill when:
- the problem is workflow-driven
- approvals or escalations matter
- handoffs are a source of pain
- roles and responsibilities across steps are unclear
- the process must be translated into governance, data, or technical implications

## Inputs
The skill works best with:
- process objective
- trigger
- participants
- current steps
- pain points
- decisions or approvals
- exceptions
- desired outcome

## Process

### 1. Clarify process scope
Identify:
- where the process starts
- where it ends
- what is in scope
- what is out of scope

### 2. Identify participants
List:
- actors
- roles
- teams
- systems if relevant

### 3. Identify flow structure
Clarify:
- start event
- main tasks
- decision points
- handoffs
- end states
- exception paths

### 4. Identify pain points
Highlight:
- delays
- ambiguity
- redundant approvals
- missing ownership
- unclear exception handling

### 5. Identify downstream implications
Consider:
- governance implications
- data implications
- technical/workflow automation implications

## Output
The skill should generally return:
- process objective
- process description in plain language
- BPMN-style participant/flow structure
- key decision points
- pain points and weaknesses
- improvement opportunities
- downstream governance/data/technical implications

## Output formats
Depending on context, the skill may output:
- BPMN-style textual structure
- Mermaid flowchart or swimlane-style approximation
- structured diagram brief for BPMN tooling

## Special rule
Prefer clarity and usefulness over strict notation purity when strict BPMN detail adds no value.
