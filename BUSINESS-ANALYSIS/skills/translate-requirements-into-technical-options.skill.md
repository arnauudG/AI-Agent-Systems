# Skill: Translate Requirements into Technical Options

## Purpose
Translate business requirements into realistic technical implementation options without collapsing prematurely into a single over-specified solution.

## Use when
Use this skill when:
- structured requirements exist
- multiple implementation paths are possible
- downstream engineering teams need options
- the business need must be connected to technical architecture direction

## Inputs
The skill works best with:
- problem statement
- desired outcomes
- stakeholders
- structured requirements
- constraints
- current systems or platform context
- MVP scope if defined

## Process

### 1. Reconfirm what must be achieved
Clarify the outcome before discussing technology.

### 2. Identify technical implications
Determine:
- what must be built, changed, integrated, or exposed
- what type of system behavior is needed
- what scale, security, traceability, or workflow implications exist

### 3. Generate plausible option types
For example:
- workflow vs metadata support
- batch vs event-driven delivery
- manual-plus-governed process vs automation
- light integration vs deeper platform integration
- narrow MVP path vs more strategic long-term path

### 4. Compare options pragmatically
Evaluate each option against:
- business fit
- delivery speed
- complexity
- ownership
- supportability
- stage appropriateness
- future extensibility where relevant

### 5. Avoid fake certainty
If constraints do not justify a single solution yet, keep options open and explicit.

## Output
The skill should generally return:
- requirement interpretation
- technical implications
- option set
- trade-off comparison
- recommended option
- deferred decisions
- assumptions and risks
