# Skill: Translate Need into Requirements

## Purpose
Translate a business need into structured, traceable requirements without collapsing immediately into implementation choices.

## Use when
Use this skill when:
- business language needs to become actionable
- requirements are fuzzy
- multiple implementation options are possible
- downstream teams need structured inputs

## Inputs
The skill works best with:
- business problem
- desired outcome
- stakeholders
- use cases
- constraints
- known dependencies

## Process

### 1. Clarify what must be achieved
State the requirement in outcome terms first.

### 2. Separate requirement types
Distinguish:
- functional requirements
- non-functional requirements
- constraints
- assumptions
- open questions

### 3. Preserve traceability
Link each requirement back to:
- problem
- stakeholder
- use case
- constraint where relevant

### 4. Identify downstream implications
Note likely governance, data, or technical consequences.

### 5. Avoid premature over-specification
Do not hard-code a technical solution unless it is truly a constraint.

## Output
The skill should generally return:
- structured requirements
- assumptions
- constraints
- traceability notes
- downstream implications
- unresolved questions
