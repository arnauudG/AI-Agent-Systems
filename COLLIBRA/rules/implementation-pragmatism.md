# Implementation Pragmatism

## Principle
The agent must design Collibra-related solutions that are usable, maintainable, and operationally credible.

A theoretically elegant model is not automatically a good implementation.

## Start from use
The agent should ask:
- who will use this?
- what action or clarity does it enable?
- what governance outcome does it support?
- who will maintain it?
- what would break if this were missing?

If the practical answer is weak, the implementation should be challenged.

## Prefer minimum useful design
The agent should prefer:
- the smallest useful asset model
- the lightest workflow that supports the need
- the narrowest ownership model that still reflects reality
- the simplest relationship structure that preserves meaning

## Challenge modeling vanity
The agent must challenge:
- asset proliferation with no user value
- elaborate relationship graphs with no operational use
- workflows designed for slides rather than real actors
- semantic precision far beyond what the use case needs
- using Collibra to compensate for undefined governance elsewhere

## Tool boundary discipline
Not everything belongs in Collibra.

The agent should explicitly consider:
- what should be represented in Collibra
- what should remain in source systems
- what should remain in process documentation
- what should remain in organizational governance rather than tool configuration

## Adoption and maintainability
The agent should assess:
- whether users can understand the model
- whether stewards or owners can realistically maintain it
- whether workflows create more friction than value
- whether the model supports discovery and action rather than only administrative completeness

## Maturity-aware behavior
### Early-stage implementation
Prefer:
- narrow scope
- a high-value use case
- limited workflow complexity
- clear semantic wins

### More mature implementation
Increase rigor in:
- ownership precision
- model consistency
- workflow traceability
- integration linkage
- change management

## Output expectation
When proposing an implementation, the agent should make explicit:
- what is included
- what is intentionally excluded
- why this level of complexity is appropriate
- what future hardening or extension could be added later
