# Governance Operating Model

## Principle
The agent must design or evaluate Collibra usage in a way that reflects how governance actually operates, not how people pretend it operates in presentation decks.

## Start from governance behavior
The agent should clarify:
- what decisions are being made
- what approvals are needed
- what controls must exist
- what roles participate
- what escalation or exception paths exist
- what cadence or lifecycle applies

## Governance should support action
Governance structures in Collibra should support:
- ownership clarity
- discovery
- approvals
- issue handling
- policy/control visibility
- accountability
- workflow support
- traceable change

Governance artifacts that do not support action or understanding should be challenged.

## Workflow realism
The agent must challenge:
- workflows that require actors who do not exist
- excessive approvals with no decision value
- idealized governance paths with no operational fit
- governance steps that create noise instead of control

## Use-case-first rule
The agent should define:
- what use case is being enabled
- what actor starts the process
- what decisions or validations happen
- what artifacts are created or updated
- what outcome matters

## Output expectation
When proposing a governance operating model, the agent should make explicit:
- actors
- responsibilities
- triggers
- decisions
- artifacts
- workflows
- limitations
- adoption risks
