# Quick Start Prompts

Use or adapt these prompts when working with Claude.

## 1. Business framing only
Use the repository as a reasoning framework.
Lead with BUSINESS-ANALYSIS.
Help me frame the problem, identify stakeholders, clarify the desired outcome, and define a first MVP scope.
Preserve traceability and challenge vague solution-first assumptions.

---

## 2. Business need to governance design
Use the repository as a reasoning framework.
Lead with BUSINESS-ANALYSIS and COLLIBRA.
Frame the business problem, identify the key business concepts, clarify ownership and governance implications, and propose the minimum useful Collibra design.
Make trade-offs explicit and preserve traceability.

---

## 3. Business need to data MVP
Use the repository as a reasoning framework.
Lead with BUSINESS-ANALYSIS and DATA.
Frame the business problem, identify users and value, define the smallest credible data MVP, and propose the data product implications.
Make ownership, quality expectations, and accepted limitations explicit.

---

## 4. Business need to AWS implementation
Use the repository as a reasoning framework.
Lead with BUSINESS-ANALYSIS and AWS.
Frame the need briefly, then translate it into technical options on AWS.
Apply implementation pragmatism and Well-Architected thinking.
Do not overengineer the solution.

---

## 5. End-to-end translation
Use the repository as a reasoning framework.
Lead with BUSINESS-ANALYSIS, then bring in COLLIBRA, DATA, and AWS as needed.
Translate this initiative from business problem to governance implications, data implications, and technical implementation direction.
Preserve a clear translation chain and recommend the smallest valuable next step.

---

## 6. Collibra use case discovery
Use the repository as a reasoning framework.
Lead with BUSINESS-ANALYSIS and COLLIBRA.
Help me turn this vague governance idea into a concrete Collibra use case with actors, concepts, ownership, artifact implications, and workflow implications.
Challenge semantic ambiguity and governance theater.

---

## 7. Data product to AWS architecture
Use the repository as a reasoning framework.
Lead with DATA and AWS.
Take this data product idea, clarify the MVP, define the source-to-consumer flow, and propose AWS implementation options.
Keep the solution stage-aware and explicit about what is deferred.

---

## 8. BPMN / process modeling
Use the repository as a reasoning framework.
Lead with BUSINESS-ANALYSIS and apply the BPMN/process modeling guidance.
Model the process clearly with actors, triggers, main steps, decision points, exceptions, pain points, and downstream governance/data/technical implications.
Prefer clarity over notation purity.

---

## 9. Review mode
Use the repository as a reasoning framework.
Act as a reviewer.
Identify which package should lead, which rules or skills apply, and where the current proposal is weak in clarity, ownership, traceability, stage-awareness, or implementation realism.

---

## 10. Diagram-first mode
Use the repository as a reasoning framework.
Prefer diagrams where they improve clarity.
Choose the smallest useful diagram type, explain why it fits, and provide a diagram source or structured visual brief along with assumptions and omissions.
